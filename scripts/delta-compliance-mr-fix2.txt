### GS-MKT-29: Async Ticket Model for Scrape — `POST /jobs/scrape`

> As a System, I want `POST ${API_BASE_URL}/jobs/scrape` to return a ticket immediately and run the Bright Data trigger/poll loop on the worker, so that the 60s web timeout no longer bounds a scrape.

**Depends on:** backend service running, Graphile Worker active, QStash configured
**Data:** Request body `{ queries: string[], employmentTypes: string[], ownerUserId: string (from JWT) }`; response `{ ticketId: string }`; `tickets` table row `{ id, owner_user_id, job_type: "scrape", state, checkpointed_step, created_at }`
**Entry state:** Authenticated user on Step 4 activates Launch Recon after queries are finalized
**Verifiable by:** POST returns `{ ticketId }` within 500ms; worker row appears in `graphile_worker.jobs`; a scrape taking 8 minutes completes without any HTTP timeout on the client (backend PRD §8.3 Goal 2, §14 tests 4–6).
**Inherits:** GS-MKT-06, GS-RKT-29

**Acceptance Criteria:**

- The endpoint returns `{ ticketId }` synchronously — it does not block on the BD trigger or poll loop
- The BD trigger/poll loop runs exclusively on the worker, not in the web request handler
- The endpoint responds in under 500ms under normal load
- The ticket row carries the owner's `user_id` from the JWT — never from the request body
- Debit and ticket creation are atomic (per GS-RKT-29)

### GS-MKT-33: QStash Unavailability at Enqueue — 503 `QUEUE_UNAVAILABLE`

> As a System, I want `POST /jobs/scrape` to return 503 `{ error: "QUEUE_UNAVAILABLE" }` and write no ticket when QStash is unreachable, so that users get a clear failure rather than an infinite poll.

**Depends on:** GS-MKT-29
**Data:** Response body `{ error: "QUEUE_UNAVAILABLE" }` with HTTP 503; no `tickets` row inserted; no debit committed
**Entry state:** `POST /jobs/scrape` called while QStash publish fails (network error, auth error, or service outage)
**Verifiable by:** Stub QStash publish to throw → response is 503 with the documented body; no ledger row is written; no ticket row exists; client shows "Market scraping temporarily unavailable — please try again in a minute" (backend PRD §8.3, §14 test 20).
**Inherits:** GS-RKT-29

**Acceptance Criteria:**

- The debit transaction is rolled back before returning 503 — the user is not charged
- The response body exactly matches `{ error: "QUEUE_UNAVAILABLE" }`
- The client detects this specific error code and renders the user-facing copy, not a generic "something went wrong"
- The system does not write any ticket row — there is nothing for the client to poll
- The handler logs the underlying QStash error with `requestId` but does not leak it to the client

### GS-MKT-34: MARKET_PREP → MARKET Chain on Worker via `POST /claude/chain`

> As a System, I want the MARKET_PREP → MARKET chain to run sequentially on the worker with Prompt Caching active, so that long analysis runs complete reliably and cost-efficiently.

**Acceptance Criteria:**

### GS-MKT-36: Ticket Ownership Enforced on `GET /tickets/{id}`

> As a System, I want `GET /tickets/{id}` to validate that the JWT subject equals the ticket owner, so that one user cannot read another user's analysis.

**Depends on:** GS-MKT-29
**Data:** Middleware reads `req.auth.sub` from verified JWT; handler compares to `ticket.ownerUserId`; mismatch → 403 with body `{ error: "FORBIDDEN" }`
**Entry state:** Any GET to `/tickets/{id}`
**Verifiable by:** User A creates a ticket → User B GETs the same id → response is 403 with no ticket metadata in the body; audit log records the attempt (backend PRD §8.3 §14 test 13).

**Acceptance Criteria:**

- The JWT is verified before the handler runs; an unsigned or expired token returns 401
- Ownership check is performed before any ticket metadata is read or returned
- On mismatch, the response is 403 with body `{ error: "FORBIDDEN" }` and no fields revealing ticket state or existence (avoid confirming the id is valid)
- The attempt is logged with `requestId`, `sub`, `ticketId`, and the outcome
- Admin-scoped callers may bypass the check (audited via a separate code path)

### GS-MKT-37: Idempotency Key on `/jobs/scrape` and `/claude/chain`

> As a System, I want `POST /jobs/scrape` and `POST /claude/chain` to require `X-Idempotency-Key`, so that client retries do not spawn duplicate tickets or debits.

**Depends on:** GS-RKT-32; GS-MKT-29
**Data:** Header `X-Idempotency-Key: <uuid>`; Redis key `idemp:${userId}:${keyHash}` with 300s TTL
**Entry state:** Any POST to `/jobs/scrape` or `/claude/chain`
**Verifiable by:** Send the same body twice with the same key → both responses return the same `ticketId`; one ledger row; one ticket row (backend PRD §8.3).
**Inherits:** GS-RKT-32

**Acceptance Criteria:**

- Missing key → 400 `{ error: "IDEMPOTENCY_KEY_REQUIRED" }`
- The key is written to Redis only after the debit + ticket transaction commits
- A collision on an active key returns the cached `{ ticketId }` with the original HTTP status
- The 5-minute window is long enough to cover common retry scenarios but not so long that a user's later genuine retry is misclassified

## 1. Title + Classification

**Market Research (Search Query Generation, Job Scraping, Two-Phase Analysis)** — MVP

## 2. Screen

Onboarding Steps 4–5. Composite component: `AimPage.tsx` (naming debt: AimPage hosts onboarding steps 4-5).

**Component mapping:** Step 4 = `StepCollect.tsx`, Step 5 = `Step6Analysis.tsx`. Both hosted in `AimPage.tsx`.

## 3. Context

Market research is the intelligence layer that powers everything downstream. Without real job market data, resumes are generic, LinkedIn content is untargeted, and auto-apply is blind. This feature transforms the user's profile and preferences into targeted search queries, scrapes live LinkedIn job postings via Bright Data API, and runs a two-phase AI analysis to produce actionable market intelligence: target categories, compensation ranges, demand keywords, and deep-dive questions.

## 4. JTBD (Jobs To Be Done)

> When I'm exploring the job market, I want to see what's actually available for someone like me, so I can focus my effort on realistic opportunities instead of guessing.

> When I see market data, I want to understand compensation ranges and demand signals, so I can set informed expectations and negotiate from strength.

> When the AI suggests target categories, I want to understand why each one fits me, so I can make confident decisions about where to aim.

## 5. Emotional Framing

- **Entry**: Curious anticipation — "What does the market actually look like for me?" The user is about to get real data about their worth and opportunities.
- **During**: Discovery and validation — watching real jobs stream in feels like reconnaissance. The analysis phase should feel like getting insider intelligence. Category cards with "why you fit" rationale should trigger "yes, that's exactly right" moments.
- **Exit**: Informed and strategic — "I know exactly where I stand and where to aim." The user leaves with a clear mental model of their market position.

## 6. Goals

- Generate 3-6 relevant search queries from any professional profile within 10 seconds
- Scrape 50+ deduplicated job postings per search session (varies by market)
- Two-phase analysis completes in under 60 seconds
- Produce at least 2 actionable target categories with compensation ranges for every user
- Zero manual data entry — everything derived from resume + preferences

## 7. Assumptions

- Bright Data LinkedIn Jobs Scraper returns sufficient results for most professional searches (50+ jobs)
- LinkedIn job postings contain enough structured data for meaningful analysis
- Two-phase analysis (MARKET_PREP → MARKET) produces higher quality than single-phase for datasets > 10 jobs
- BD API availability is 99%+ during business hours
- Annual salaries from BD are reliable; hourly rates must be extracted from description text via regex
- Employment type cycling across queries produces more diverse results than sending all types per query
- Job data is available within the user's session wait tolerance (BD polling completes within 6 minutes)

## 8. Feature Description

Market Research is a continuous two-step process that transforms a professional profile into actionable job market intelligence. The user experiences it as a single flow: recon → analysis → targets.

**Step 4: Search Query Generation & Job Scraping**

Claude AI (QUERY_GEN prompt) generates 3-6 LinkedIn search queries from the user's profile, preferences, and avoid terms. Queries are presented as editable search vectors — the user can modify, add, or remove queries before launching. No-fly zones (avoid terms from the profile) are displayed as removable tags.

A "Launch Recon" button triggers parallel job scraping via the Bright Data LinkedIn Jobs Scraper API. The system cycles employment types across queries for balanced coverage. A progress screen shows 5-step animation during the 3-5 minute scrape. The system polls every 30 seconds and forces partial results after 3 minutes if some snapshots are still pending.

Results are deduplicated by title+company key. Each job listing is normalized to a standard shape: title, company, location, description, salary, seniority, employment type, Easy Apply status, URL, industries, job function, applicant count, posted date.

Query performance stats show how many results each query yielded, helping users understand which vectors are most productive.

**Step 4→5 Transition: Seamless Analysis (same screen)**

After scrape completes, the analysis pipeline runs automatically on the **same screen** — the user sees the scrape progress messages transition directly into analysis progress messages ("Scanning for keywords...", "Mapping comp ranges...", "Identifying target categories...", "Building targeting dossier...") without a page change. When analysis completes, an "Analysis Complete" CTA appears with summary stats (category count, comp range, jobs analyzed). The user taps "View Your Targets" to proceed to category selection.

**CRITICAL: The entire recon sweep (query generation, job scraping, AND two-phase analysis) completes on ONE screen before the user ever sees the categories page.** The user does not navigate away during any part of the recon sweep. Categories are a separate page/view that appears only AFTER all recon processing is done.

**Implementation:** AimPage uses a 3-substep architecture. StepCollect stays mounted during analysis and displays progress via props. Step6Analysis runs hidden in `display: none` and reports status via `onStatusChange` callback. See FLOW_SPEC.md for the full substep table.

**Step 5: Category Selection & Lock (separate view — after recon completes)**

Target categories are presented as selectable cards showing: category name, compensation range, job volume, match strength badge (top candidate / strong fit / worth exploring), and an expandable "why you fit" rationale. Categories are displayed in the order the AI returns them (best fit first) — array position is the sort order, no ranking UI.

Users toggle categories on/off. Deselected categories appear grayed out with strikethrough. Discovery recommendations allow adding up to 10 total categories. A FOMO warning shows missed competitiveness percentage if not all categories are selected.

"Lock Your Targets" writes `rankedCategories = jobTypes.filter(jt => jt.selected)` to session and advances to the **Dashboard** (target state — baseline competitiveness score becomes visible on dashboard entry; Deep-Dive Q&A is a dashboard activity the user can opt into from there). Spec-ahead-of-code: shipped code advances directly to Step 6 (Deep Dive Q&A) as the next onboarding step.

A "Refresh Analysis" button re-runs the two-phase analysis on existing job data without re-scraping.

If fewer than 10 jobs were returned, an advisory suggests broadening search parameters (adding employment types, expanding location radius, relaxing avoid terms).

### v3 — Async ticket model (preserves locked single-screen UX)

**The locked constraint is preserved:** the entire recon sweep still completes on ONE screen before the user sees categories. What changes is the backend — the 3-5 min scrape + two-phase chain now runs on the Fly worker (escaping the 60s Vercel timeout), with the frontend polling a ticket.

Per plan D2 (hybrid):

- **Primary path (identical UX to v2):** User clicks "Launch Recon" → frontend calls `POST ${API_BASE_URL}/jobs/scrape` (returns `{ticketId}`) → StepCollect stays mounted → polls `GET ${API_BASE_URL}/tickets/{id}` every 3-5s → progress messages driven by ticket.progress.stage (`BD_TRIGGER`, `BD_POLL`, `BD_RESULTS`, `MARKET_PREP_INPUT` …). Categories page appears only when ticket is `done`.
- **Tab-close fallback (new in v3):** If the user closes the tab mid-scrape, the worker continues server-side. `ticketId` is written to SessionData so on return the frontend can either (a) resume polling if still running, showing a "Your scrape is still running…" banner; or (b) load the `done` result. **The locked single-screen flow is preserved for the normal path; graceful recovery is added for tab close.**
- **QStash unavailability at enqueue:** `POST /jobs/scrape` returns `503 { error: "QUEUE_UNAVAILABLE" }` and no ticket is written — frontend surfaces "Market scraping temporarily unavailable" rather than an infinite polling loop (per backend PRD §8.4).
- **Two-phase chain:** MARKET_PREP → MARKET now execute on the worker via `POST ${API_BASE_URL}/claude/chain`, sequentially with **Prompt Caching active** on each (mandatory per backend PRD §8.16).
- **Idempotency:** `POST /jobs/scrape` requires `X-Idempotency-Key` header; same key within 5 min returns same ticketId without a second debit.
- **Ticket ownership:** `GET /tickets/{id}` validates `JWT.sub === ticket.ownerUserId`; mismatch → 403. Prevents other users from polling another's recon.

3. Add a new query manually → verify it's included in the scrape
4. Launch recon → verify progress screen shows during scrape → verify results arrive
5. Verify job deduplication (same title+company from different queries = 1 listing)

13. **BD discovery-mode regression test (RT-014/RT-015):** intercept the BD trigger URL emitted by the worker; assert it contains `type=discover_new&discover_by=keyword`. Run a real scrape with a narrow query ("LLM Product Lead") and assert the response is NOT a 1-element array of `{error_code: "dead_page"}` records. If either fails, the scraper has regressed to the dataset's default single-page collect behavior.

## 15. Out of Scope

- Real-time job alerts or saved searches
- Non-LinkedIn job sources (Indeed, Glassdoor, etc.)
- Salary negotiation recommendations
- Company-specific research (culture, reviews, interview process)
- Job bookmarking or favoriting

## 16. Decisions

1. **Manual URL input**: LinkedIn-only for MVP with URL validation. Post-MVP: generic job posting URL scraping.
2. **v3: Async ticket model preserves locked single-screen UX.** Primary path keeps StepCollect mounted + polls ticket; tab-close fallback resumes on return via ticketId in SessionData. (Per plan D2 hybrid; backend PRD §8.4.)
3. **v3: Prompt Caching is mandatory on QUERY_GEN, MARKET_PREP, MARKET.** Cache hit rate alert if <80% over 1h — indicates canonical context drift. (Per backend PRD §8.16.)
4. **v3: X-Idempotency-Key on every `/jobs/scrape` and `/claude/chain` call.** Prevents double-debit on client retries. (Per backend PRD §8.4.)
5. **v3: wrapUntrustedData() is a CI-enforced contract** on any Claude call that processes BD data in the worker — not convention. Prompt injection defense per backend PRD §8.17. Gray Swan attack rates (1.4% direct, ~50% indirect per doc) make this non-negotiable.
2. **Market refresh pricing**: Free for re-analysis of same data. 50 rockets if user re-scrapes new data.
3. **Thin data threshold**: Show advisory at < 15 jobs. Block analysis at < 5 jobs with suggestion to broaden search parameters.

## Async Ticket Contract (v3)

Not a user-facing form input, but the contract that the Launch Recon action now produces.

### `POST ${API_BASE_URL}/jobs/scrape`

- **Headers (required):**
  - `Authorization: Bearer <JWT>`
  - `X-Idempotency-Key: <UUIDv4>` — client regenerates for each distinct Launch, reuses on retry
  - `Content-Type: application/json`
- **Body:** `{ queries: string[], employmentTypes: string[] }` — `ownerUserId` is taken from the JWT, not the body
- **Success response:** `{ ticketId: string }` with HTTP 201
- **Queue-unavailable response:** HTTP 503 `{ error: "QUEUE_UNAVAILABLE" }` — no debit, no ticket
- **Client follow-up:** write `ticketId` to `SessionData.activeTicketId`, then poll `GET ${API_BASE_URL}/tickets/{id}` every 3–5s (jittered) until terminal state

### `POST ${API_BASE_URL}/claude/chain` (MARKET_PREP → MARKET)

- **Headers:** same as `/jobs/scrape`
- **Body:** `{ steps: [{ prompt: "MARKET_PREP" }, { prompt: "MARKET" }], marketRawTicketId?: string }`
- **Success response:** `{ ticketId: string }` with HTTP 201
- **Prompt Caching:** server-side only — caller does not set cache headers; the worker applies `cache_control: ephemeral` on every Claude call

### `GET ${API_BASE_URL}/tickets/{id}`

- **Auth:** JWT required; server validates `JWT.sub === ticket.ownerUserId`; mismatch → 403 `{ error: "FORBIDDEN" }`
- **Response shape:** `{ id, state: "queued"|"running"|"done"|"failed"|"cancelled", progressStage: string|null, result?: unknown, error?: string }`
- **Polling:** client stops immediately on terminal state; result payload is merged into SessionData

### Tab-close recovery

On StepCollect mount, if `SessionData.activeTicketId` is non-null, the client reads the ticket — if non-terminal, polling resumes; if terminal, the result is loaded and rendered in place without a re-trigger.

  try {

// ── MARKET output parsing ─────────────────────────────────────────────────────

interface MarketRawOutput {
  keywords?: Keyword[];
  compRanges?: { low: string; median: string; high: string } | null;
  jobTypes?: Array<{
    name: string;
    description: string;
    why: string;
    compRange?: string | null;
    volume?: string | null;
    matchStrength?: "high" | "mid" | "low";
    searchTerms?: string[];
  }>;
  miningQuestions?: Array<{ id: string; question: string; why?: string }>;
  discoveryRecs?: Array<{
    id: string;
    name: string;
    rationale: string;
    compRange?: string;
    volume?: string;
  }>;
  exclusionTags?: string[];
  educationVisibility?: string;
}

function parseMarketOutput(raw: string): {
  marketAnalysis: MarketAnalysis;
  jobTypes: JobType[];
  miningQuestions: Array<{ id: string; question: string; why?: string }>;
} {
  const parsed = JSON.parse(cleanJson(raw)) as MarketRawOutput;

  const jobTypes: JobType[] = (parsed.jobTypes ?? []).map((jt) => ({
    name: jt.name ?? "",
    description: jt.description ?? "",
    why: jt.why ?? "",
    compRange: jt.compRange ?? null,
    volume: jt.volume ?? null,
    matchStrength: jt.matchStrength ?? "mid",
    selected: true, // GS-MKT-14: all default selected
    searchTerms: jt.searchTerms,
  }));

  const marketAnalysis: MarketAnalysis = {
    keywords: parsed.keywords ?? [],
    compRanges: parsed.compRanges ?? null,
    discoveryRecs: (parsed.discoveryRecs ?? []).map((r, i) => ({
      id: r.id ?? `rec-${i}`,
      name: r.name ?? "",
      rationale: r.rationale ?? "",
      compRange: r.compRange,
      volume: r.volume,
    })),
    exclusionTags: parsed.exclusionTags ?? [],
    educationVisibility: parsed.educationVisibility ?? "show",
  };

  return {
    marketAnalysis,
    jobTypes,
    miningQuestions: parsed.miningQuestions ?? [],
  };
}

// ── FOMO competitiveness delta ────────────────────────────────────────────────

function getFomoDelta(jobTypes: JobType[]): number {
  const allSelected = jobTypes.every((jt) => jt.selected);
  if (allSelected) return 0;
  const unselected = jobTypes.filter((jt) => !jt.selected);
  // Each unselected category represents a fraction of the category weight
  const perCat = WEIGHTS.categoriesSelected / Math.max(jobTypes.length, 1);
  return Math.round(unselected.length * perCat);
}

// ── Match strength badge label/color ─────────────────────────────────────────

function matchBadge(strength: "high" | "mid" | "low"): {
  label: string;
  color: string;
} {
  switch (strength) {
    case "high":
      return { label: "Top Candidate", color: "var(--primary, #ff5a17)" };
    case "mid":
      return { label: "Strong Fit", color: "var(--warning, #eab308)" };
    case "low":
      return { label: "Worth Exploring", color: "var(--text-muted)" };
  }
}

// AimPage.tsx — Onboarding Steps 4-5 orchestrator
// Feature: market-research | PRD Section 8, FLOW_SPEC substep architecture
// BUG-004: page.tsx already renders AimPage — this is the component, not the route
// BUG-012: fresh loadSession() each handler
// HYGIENE Rule 29: saveSession before substep advance
// BUG-064: substep state single source of truth
// BUG-074: client never writes currentStep > maxStep+1
// GS-MKT-20: auto-save callbacks omit jobTypes; only Lock includes them

import React, { useEffect, useRef, useState, useCallback } from "react";
import { loadSession, saveSession } from "@/lib/storage";
import { StepCollect } from "@/components/steps/StepCollect";
import { Step6Analysis } from "@/components/steps/Step6Analysis";
import type { SessionData } from "@/lib/types";

// ── Substep architecture ──────────────────────────────────────────────────────
// substep 0: StepCollect — query editor + scrape launch + scrape progress
// substep 1: StepCollect (analysis-running mode) + Step6Analysis hidden
// substep 2: Step6Analysis (categories view)

type Substep = 0 | 1 | 2;

export function AimPage() {
  const [session, setSession] = useState<SessionData | null>(null);
  const [loading, setLoading] = useState(true);
  const [substep, setSubstep] = useState<Substep>(0);

  // Analysis status reported by Step6Analysis to show in StepCollect
  const [analysisStatus, setAnalysisStatus] = useState<
    "idle" | "running" | "done" | "error"
  >("idle");
  const [analysisError, setAnalysisError] = useState<string | undefined>(
    undefined,
  );
  const [analysisJobCount, setAnalysisJobCount] = useState<number | undefined>(
    undefined,
  );

  const saveInProgress = useRef(false);

  // ── Load session on mount ─────────────────────────────────────────────────

  useEffect(() => {
    loadSession()
      .then((s) => {
        const loaded = s ?? ({} as SessionData);
        setSession(loaded);

        // Determine initial substep from session state
        if (loaded.marketAnalysis && loaded.jobTypes?.length) {
          // Analysis already done — show categories
          setSubstep(2);
          setAnalysisStatus("done");
        } else if (loaded.marketRaw) {
          // Scrape done, analysis not yet complete — trigger analysis
          setSubstep(1);
        } else {
          // Fresh entry — query editor
          setSubstep(0);
        }
      })
      .catch(() => {
        setSession({} as SessionData);
        setSubstep(0);
      })
      .finally(() => setLoading(false));
  }, []);

  // ── Session update helper (BUG-012: always read fresh) ───────────────────

  const handleSessionUpdate = useCallback(
    async (data: Partial<SessionData>) => {
      if (saveInProgress.current) return;
      saveInProgress.current = true;
      try {
        // BUG-012: fresh loadSession each time
        const latest = (await loadSession()) ?? ({} as SessionData);
        const updated = { ...latest, ...data };
        await saveSession(updated);
        setSession(updated);
      } finally {
        saveInProgress.current = false;
      }
    },
    [],
  );

  // ── Scrape complete callback ───────────────────────────────────────────────
  // Called by StepCollect when BD polling finishes.
  // HYGIENE Rule 29: saveSession BEFORE advancing substep.

  const handleScrapeComplete = useCallback(
    async (data: {
      marketRaw: string;
      queryStats: Array<{ query: string; resultCount: number }>;
      marketSource: "api";
    }) => {
      // Save scrape results before advancing
      const latest = (await loadSession()) ?? ({} as SessionData);
      const updated: SessionData = {
        ...latest,
        marketRaw: data.marketRaw,
        queryStats: data.queryStats,
        marketSource: data.marketSource,
      };
      await saveSession(updated);
      setSession(updated);

      // HYGIENE Rule 29: save complete, now advance substep
      setSubstep(1);
      setAnalysisStatus("idle");
    },
    [],
  );

  // ── Analysis status callback ───────────────────────────────────────────────
  // Step6Analysis reports status so StepCollect can render the progress overlay.

  const handleAnalysisStatusChange = useCallback(
    (
      status: "idle" | "running" | "done" | "error",
      error?: string,
      jobCount?: number,
    ) => {
      setAnalysisStatus(status);
      setAnalysisError(error);
      setAnalysisJobCount(jobCount);
    },
    [],
  );

  // ── Analysis session update ───────────────────────────────────────────────
  // Step6Analysis saves intermediate results (marketPrepReport, marketAnalysis).
  // NOTE: does NOT include jobTypes (GS-MKT-20).

  const handleAnalysisSessionUpdate = useCallback(
    async (data: Partial<SessionData>) => {
      await handleSessionUpdate(data);
    },
    [handleSessionUpdate],
  );

  // ── View targets callback ─────────────────────────────────────────────────
  // When analysis is done and user taps "View Your Targets", advance substep.
  // HYGIENE Rule 29: session already saved by analysis pipeline; just advance.

  const handleViewTargets = useCallback(() => {
    setSubstep(2);
  }, []);

  // ── Lock categories (complete) ────────────────────────────────────────────
  // Step6Analysis calls onComplete with jobTypes + rankedCategories + currentStep.
  // HYGIENE Rule 29: save BEFORE state update.
  // GS-MKT-20: hardcoded currentStep: 6 (HYGIENE Rule 56).
  // BUG-074: never write currentStep > maxStep+1.

  const handleComplete = useCallback(async (data: Partial<SessionData>) => {
    const latest = (await loadSession()) ?? ({} as SessionData);
    const currentMax = latest.maxStep ?? latest.currentStep ?? 0;
    const nextStep = data.currentStep ?? 6;
    // BUG-074: clamp advancement
    const safeStep = Math.min(nextStep, currentMax + 1);

    const updated: SessionData = {
      ...latest,
      ...data,
      currentStep: safeStep,
      maxStep: Math.max(currentMax, safeStep),
    };

    // HYGIENE Rule 29: save before page re-init
    await saveSession(updated);
    setSession(updated);

    // Trigger page.tsx re-render at new step (self-managing page pattern)
    window.location.reload();
  }, []);

  // ── Render ────────────────────────────────────────────────────────────────

  if (loading || !session) {
    return (
      <div
        style={{
          display: "flex",
          alignItems: "center",
          justifyContent: "center",
          minHeight: "60vh",
          color: "var(--text-muted)",
          fontSize: "14px",
        }}
      >
        Loading...
      </div>
    );
  }

  const containerStyle: React.CSSProperties = {
    minHeight: "100vh",
    background: "var(--background)",
    color: "var(--text)",
    fontFamily: "var(--font-body)",
  };

  // Step6Analysis is mounted hidden in substeps 1-2 so it can run the pipeline
  // while StepCollect shows the progress overlay (PRD Section 8 implementation).
  const step6Visible = substep === 2;

  return (
    <div style={containerStyle}>
      {/* StepCollect — visible in substeps 0 and 1 */}
      {substep <= 1 && (
        <StepCollect
          session={session}
          substep={substep}
          analysisStatus={substep === 1 ? analysisStatus : "idle"}
          analysisError={substep === 1 ? analysisError : undefined}
          jobCount={analysisJobCount}
          onScrapeComplete={handleScrapeComplete}
          onViewTargets={handleViewTargets}
          onSessionUpdate={(data) => void handleSessionUpdate(data)}
        />
      )}

      {/* Step6Analysis — mounted (hidden) in substep 1 to run pipeline,
          fully visible in substep 2 (categories view) */}
      {substep >= 1 && (
        <div style={{ display: step6Visible ? "block" : "none" }}>
          <Step6Analysis
            session={session}
            initialViewPhase={substep === 1 ? "analyzing" : "categories"}
            onStatusChange={handleAnalysisStatusChange}
            onComplete={(data) => void handleComplete(data)}
            onSessionUpdate={(data) => void handleAnalysisSessionUpdate(data)}
          />
        </div>
      )}
    </div>
  );
}

async function checkAndIncrementBudget(): Promise<{
  allowed: boolean;
  used: number;
}> {
  const redis = getRedis();
  if (!redis) {
    // No Redis configured — allow (dev mode)
    return { allowed: true, used: 0 };
  }
  const key = dailyBudgetKey();
  const current = (await redis.get<number>(key)) ?? 0;
  if (current >= DAILY_BD_BUDGET) {
    return { allowed: false, used: current };
  }
  await redis.incr(key);
  // Expire at end of day
  await redis.expire(key, 86400);
  return { allowed: true, used: current + 1 };
}

// ── Rate limiting (10 req/min per IP) ────────────────────────────────────────

function getRatelimit(): Ratelimit | null {
  const redis = getRedis();
  if (!redis) return null;
  return new Ratelimit({
    redis,
    limiter: Ratelimit.slidingWindow(10, "1 m"),
    prefix: "bd:rl",
  });
}

/**
 * Trigger one BD discovery snapshot for a single keyword+employmentType combo.
 * BUG-031: BD uses keyword input, NOT LinkedIn search URLs.
 */
async function triggerSnapshot(
  keyword: string,
  location: string,
  employmentType: string,
  apiKey: string,
): Promise<string> {
  const url = `${BD_API_BASE}/datasets/v3/trigger?dataset_id=${BD_DATASET_ID}&type=discover_new&discover_by=keyword&limit_per_input=30&format=json&include_errors=true`;

  const body = JSON.stringify([
    {
      keyword,
      location: location || "United States",
      country: "US",
      job_type: employmentType,
    },
  ]);

  const res = await fetch(url, {
    method: "POST",
    headers: {
      Authorization: `Bearer ${apiKey}`,
      "Content-Type": "application/json",
    },
    body,
  });

  if (!res.ok) {
    const text = await res.text().catch(() => "");
    throw new Error(`BD trigger failed (${res.status}): ${text.slice(0, 200)}`);
  }

  const data = (await res.json()) as BDTriggerResponse;
  return data.snapshot_id;
}

/**
 * Check snapshot status.
 */
async function checkSnapshot(
  snapshotId: string,
  apiKey: string,
): Promise<BDProgressResponse> {
  const url = `${BD_API_BASE}/datasets/v3/snapshot/${snapshotId}/progress`;
  const res = await fetch(url, {
    headers: { Authorization: `Bearer ${apiKey}` },
  });
  if (!res.ok) {
    throw new Error(`BD progress check failed (${res.status})`);
  }
  return (await res.json()) as BDProgressResponse;
}

/**
 * Download snapshot data.
 */
async function downloadSnapshot(
  snapshotId: string,
  apiKey: string,
): Promise<Array<Record<string, unknown>>> {
  const url = `${BD_API_BASE}/datasets/v3/snapshot/${snapshotId}?format=json`;
  const res = await fetch(url, {
    headers: { Authorization: `Bearer ${apiKey}` },
  });
  if (!res.ok) {
    throw new Error(`BD download failed (${res.status})`);
  }
  try {
    const data = await res.json();
    return Array.isArray(data) ? data : [];
  } catch {
    return [];
  }
}

/**
 * Deduplicate jobs by title+company (case-insensitive).
 * Also tracks per-query result counts (before dedup).
 */
function deduplicateJobs(
  jobsByQuery: Array<{ query: string; jobs: JobListing[] }>,
): { jobs: JobListing[]; queryStats: QueryStat[] } {
  const seen = new Set<string>();
  const deduped: JobListing[] = [];
  const queryStats: QueryStat[] = [];

  for (const { query, jobs } of jobsByQuery) {
    queryStats.push({ query, resultCount: jobs.length });
    for (const job of jobs) {
      const key = `${job.title.toLowerCase().trim()}|${job.company.toLowerCase().trim()}`;
      if (!seen.has(key) && (job.title || job.company)) {
        seen.add(key);
        deduped.push(job);
      }
    }
  }

  return { jobs: deduped, queryStats };
}

// ── Cycle employment types across queries ─────────────────────────────────────

function cycleEmploymentTypes(
  queries: string[],
  employmentTypes: string[],
): Array<{ query: string; employmentType: string }> {
  if (!employmentTypes || employmentTypes.length === 0) {
    return queries.map((q) => ({ query: q, employmentType: "Full-time" }));
  }
  const mapped = employmentTypes.map(
    (et) => EMPLOYMENT_TYPE_MAP[et] ?? "Full-time",
  );
  return queries.map((q, i) => ({
    query: q,
    employmentType: mapped[i % mapped.length],
  }));
}

// ── POST handler ──────────────────────────────────────────────────────────────

export async function POST(req: NextRequest) {
  // HYGIENE Rule 27: validateOrigin as boolean if-check, never try/catch
  if (!validateOrigin(req)) {
    return NextResponse.json({ error: "Forbidden" }, { status: 403 });
  }

  // Rate limit: 10 requests per minute per IP (GS-MKT-10)
  const ip =
    req.headers.get("x-forwarded-for")?.split(",")[0]?.trim() ?? "unknown";
  const ratelimit = getRatelimit();
  if (ratelimit) {
    const { success, reset } = await ratelimit.limit(ip);
    if (!success) {
      return NextResponse.json(
        { error: "Too many requests. Please wait and try again." },
        {
          status: 429,
          headers: {
            "Retry-After": String(Math.ceil((reset - Date.now()) / 1000)),
          },
        },
      );
    }
  }

  const apiKey = process.env.BRIGHTDATA_API_KEY;

  let body: Record<string, unknown>;
  try {
    body = (await req.json()) as Record<string, unknown>;
  } catch {
    return NextResponse.json(
      { error: "Invalid request body" },
      { status: 400 },
    );
  }

  const action = body.action as string;

  // ── TRIGGER ──────────────────────────────────────────────────────────────

  if (action === "trigger") {
    // RT-201: auth guard — reject unauthenticated trigger requests
    const authToken = await getAuthToken();
    if (!authToken || !verifyJWT(authToken)) {
      return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
    }

    // Check daily budget (GS-MKT-10)
    const { allowed, used } = await checkAndIncrementBudget();
    if (!allowed) {
      return NextResponse.json(
        {
          error: `Daily scrape budget exhausted (${used}/${DAILY_BD_BUDGET} requests used today). Try again tomorrow.`,
        },
        { status: 429 },
      );
    }

    const queries = body.queries as string[] | undefined;
    const location = (body.location as string) || "United States";
    const employmentTypes = (body.employmentTypes as string[]) || ["Full-time"];

    if (!queries || !Array.isArray(queries) || queries.length === 0) {
      return NextResponse.json(
        { error: "At least one search query is required" },
        { status: 400 },
      );
    }

    // Dev/test mode — no real API key
    if (!apiKey) {
      // Return mock snapshot IDs for local development
      const snapshotIds = queries.map((_, i) => `mock-snapshot-${i}`);
      const queryMap: Record<string, string> = {};
      queries.forEach((q, i) => {
        queryMap[snapshotIds[i]] = q;
      });
      return NextResponse.json({ snapshotIds, queryMap });
    }

    const queryEmploymentPairs = cycleEmploymentTypes(queries, employmentTypes);

    const snapshotIds: string[] = [];
    const queryMap: Record<string, string> = {};
    const errors: string[] = [];

    // Trigger one snapshot per query (parallel)
    await Promise.allSettled(
      queryEmploymentPairs.map(async ({ query, employmentType }) => {
        try {
          const snapshotId = await triggerSnapshot(
            query,
            location,
            employmentType,
            apiKey,
          );
          snapshotIds.push(snapshotId);
          queryMap[snapshotId] = query;
        } catch (e) {
          errors.push(
            e instanceof Error ? e.message : `Trigger failed for: ${query}`,
          );
        }
      }),
    );

    if (snapshotIds.length === 0) {
      return NextResponse.json(
        { error: `All scrape triggers failed: ${errors.join("; ")}` },
        { status: 502 },
      );
    }

    return NextResponse.json({
      snapshotIds,
      queryMap,
      warnings: errors.length > 0 ? errors : undefined,
    });
  }

  // ── POLL ──────────────────────────────────────────────────────────────────

  if (action === "poll") {
    const snapshotIds = body.snapshotIds as string[] | undefined;
    const queryMap = (body.queryMap as Record<string, string>) || {};
    const force = body.force === true;

    if (
      !snapshotIds ||
      !Array.isArray(snapshotIds) ||
      snapshotIds.length === 0
    ) {
      return NextResponse.json(
        { error: "snapshotIds required for poll" },
        { status: 400 },
      );
    }

    // RT-202: validate snapshot ID format before using in URL construction
    const snapshotIdRegex = /^[a-zA-Z0-9_-]{8,64}$/;
    for (const sid of snapshotIds) {
      if (!snapshotIdRegex.test(sid)) {
        return NextResponse.json(
          { error: "Invalid snapshot ID format" },
          { status: 400 },
        );
      }
    }

    // Dev mock mode
    if (!apiKey) {
      // Return mock results
      const mockJobs: JobListing[] = snapshotIds.flatMap((sid) => {
        const query = queryMap[sid] ?? "mock query";
        return [
          normalizeJob(
            {
              title: `Mock Job for ${query}`,
              company: "Mock Corp",
              location: "Remote",
              description: "Mock job description for development testing.",
              employment_type: "Full-time",
            },
            query,
          ),
        ];
      });
      const queryStats = snapshotIds.map((sid) => ({
        query: queryMap[sid] ?? sid,
        resultCount: 1,
      }));
      return NextResponse.json({
        done: true,
        jobs: mockJobs,
        total: mockJobs.length,
        pending: 0,
        queryStats,
      });
    }

    // Check progress for all snapshots in parallel
    const progressResults = await Promise.allSettled(
      snapshotIds.map((sid) => checkSnapshot(sid, apiKey)),
    );

    const readyIds: string[] = [];
    const pendingIds: string[] = [];

    progressResults.forEach((result, i) => {
      const sid = snapshotIds[i];
      if (result.status === "fulfilled") {
        const { status } = result.value;
        if (status === "ready") {
          readyIds.push(sid);
        } else if (status === "failed") {
          // Treat failed snapshots as done (skip them)
        } else {
          pendingIds.push(sid);
        }
      } else {
        // Error checking progress — treat as pending
        pendingIds.push(sid);
      }
    });

    const allDone = pendingIds.length === 0;
    const shouldReturn = allDone || force;

    if (!shouldReturn) {
      return NextResponse.json({
        done: false,
        pending: pendingIds.length,
        total: snapshotIds.length,
      });
    }

    // Download ready snapshots
    const jobsByQuery: Array<{ query: string; jobs: JobListing[] }> = [];

    await Promise.allSettled(
      readyIds.map(async (sid) => {
        const query = queryMap[sid] ?? sid;
        try {
          const rawJobs = await downloadSnapshot(sid, apiKey);
          const normalized = rawJobs.map((r) => normalizeJob(r, query));
          jobsByQuery.push({ query, jobs: normalized });
        } catch {
          jobsByQuery.push({ query, jobs: [] });
        }
      }),
    );

    // Add zero-result entries for snapshots that weren't ready (forced partial)
    for (const sid of pendingIds) {
      const query = queryMap[sid] ?? sid;
      const alreadyHasEntry = jobsByQuery.some((jbq) => jbq.query === query);
      if (!alreadyHasEntry) {
        jobsByQuery.push({ query, jobs: [] });
      }
    }

    const { jobs, queryStats } = deduplicateJobs(jobsByQuery);

    return NextResponse.json({
      done: true,
      jobs,
      total: jobs.length,
      pending: pendingIds.length,
      queryStats,
    });
  }

  return NextResponse.json(
    { error: `Unknown action: ${action}` },
    { status: 400 },
  );
}
