## 3. Context

Onboarding is the foundation of the entire pipeline. Every downstream feature — market research, resume generation, LinkedIn content, auto-apply — depends on the data collected here. A user who abandons onboarding produces zero value. The flow must feel fast, low-friction, and progressively rewarding: upload a resume, confirm details, set preferences, and receive an AI-generated professional profile — all in under 5 minutes.

## 4. JTBD (Jobs To Be Done)

> When I start using a job search tool, I want to quickly import my resume and set my preferences, so I can get personalized results without manually entering everything.

> When I see my AI-generated profile, I want to verify it understands my strengths and career direction, so I can trust the system to represent me accurately downstream.

> When I have a non-standard background (career gap, industry switch, freelance history), I want the tool to capture that context, so I can get advice that accounts for my real situation, not a generic template.

## 5. Emotional Framing

- **Entry**: Hopeful but skeptical — "Will this actually work for me?" The user has tried job tools before. They need to feel this one is different within 60 seconds of starting.
- **During**: Progressive confidence — each completed sub-step builds trust. Resume parsing should feel like magic ("it got everything right"). Preferences should feel empowering ("I'm in control"). Profile generation should feel validating ("it really understands me").
- **Exit**: Energized and ready — the celebration screen should feel like the starting gun. "My arsenal is being built. Let's go."

## 6. Goals

- User completes full onboarding (resume → preferences → profile) in under 5 minutes
- Resume parsing succeeds on 95%+ of well-formatted resumes (PDF, DOCX, TXT)
- Zero data loss — every field the user fills is persisted to encrypted session
- Profile generation produces actionable output (discipline, seniority, skills, gaps) on first attempt for 90%+ of users
- Preferences capture enough signal to generate relevant search queries downstream

## 7. Assumptions

- Users have a resume in a common format (PDF, DOCX, TXT, or plain text they can paste)
- Claude PARSE prompt handles 95%+ of well-formatted resumes without hallucinating fields
- 10MB file limit is sufficient for any reasonable resume (even image-heavy PDFs)
- Employment status (employed/left/outdated) provides enough context for downstream prompts
- Demographics are optional — the system works without them but produces better results with them
- Bidirectional hourly ↔ salary conversion at 2080 hours/year is a reasonable standard assumption
- **v3:** PARSE and PROFILE are anonymous endpoints (no user JWT required); Cloudflare Turnstile on first anonymous hit adds <100 ms transparent latency (invisible challenge). This upgrades SECURITY.md's v2 "accepted risk" to MVP hardening per research F10 / backend PRD §16 decision 17.
- **v3:** Prompt Caching on PARSE and PROFILE (system prompt + canonical context cached) — mandatory per backend PRD §8.16.

## 8. Feature Description

Onboarding is a 3-step wizard that transforms a raw resume into a structured professional profile with search preferences.

**Step 1: Resume Upload (IntroScreen)**

The user provides their resume via the IntroScreen landing page: drag-and-drop file upload (PDF, DOCX, TXT, MD — 10MB limit) or by pasting raw text. Upon upload, the system immediately triggers resume parsing as a **background task** via Claude AI (PARSE prompt) and advances the user to Step 2 (Preferences). The user does NOT wait for parsing to complete.

**v3 anonymous-endpoint hardening:** PARSE is called via `POST ${API_BASE_URL}/claude` without an auth cookie (anonymous path). On the first call from a fresh browser session, Cloudflare Turnstile issues an invisible challenge token that the frontend attaches as `CF-Turnstile-Response` header on subsequent calls. Backend validates via Siteverify API. Transparent to the user in the normal case; blocks programmatic scanners and low-effort botnets. Request per-ASN rate limit (Cloudflare) further dampens distributed abuse. Per backend PRD §8.2 layer 2 + §16 decision 17.

A persistent parsing status banner is displayed throughout Step 2:

- **During parsing**: "Parsing your resume in the background..." (yellow/olive)
- **On success**: "Resume parsed ✓" (green) — data saved to session silently
- **On failure**: "Resume parsing failed. Retry?" (red) — retry action available; does NOT block preference entry

> **Parse-failed entry state:** If resume parsing fails (timeout, unsupported format), the user sees an error toast and can retry the upload. The system does NOT block advancement to Step 2 -- preferences can be entered manually without parsed data. Only Step 3 (profile generation) gates on parsed resume data.

Session auto-saves at each sub-step boundary. If the user refreshes or returns, they resume where they left off within onboarding. If the user returns with `resumeRaw` in session but no `resumeStructured`, the system auto-triggers parsing without requiring re-upload.

> **Parallel Parse Principle**: Resume parsing and preference collection run concurrently. The user never waits idle for parsing. Questions that don't need resume data (all of Step 2) are asked while the parse runs. Only Step 3 (profile generation) requires parsed data, creating a natural synchronization point.

**Step 2: Preferences & Parameters**

All 6 substeps are **resume-independent** — they collect user intent, not resume data. No substep reads from `resumeStructured`, `personal`, or `education`. Resume parsing may complete during any substep — the banner updates but the flow is uninterrupted.

Step 2 is handled by `OnboardingPage.tsx` as 6 internal substeps: **direction** → **worktype** → **comp** → **location** → **quickcheck** → **dealbreakers**. Each substep auto-saves to session on completion.

Direction: multiselect cards, 1+ required (same/similar role, higher-level, new career, leadership, startup, something else). Default: "Same or similar role". Selecting "new career" or "something else" reveals a free-text detail field.

Work type: 3 single-select pill buttons (Full-time, Part-time, Contract). Default: Full-time. Start date: pill buttons (Immediately, 2 weeks notice, 1 month notice, 2 months notice). Default: Immediately.

Compensation: structure dropdown (default "Open to discussion"). Hourly floor and salary floor text inputs with bidirectional sync (hourly × 2080 = annual).

Location: single-select pill toggle (Remote, Hybrid, On-site). Default: Remote. Non-remote reveals city autocomplete.

Quick check: 7 tappable card-style rows (full-width boxes with checkbox indicator: filled ✓ when active, empty border when inactive). Smart defaults: US citizen ✓, authorized to work ✓, no visa sponsorship ✓, driver's license ✓, veteran ✗, disability ✗, security clearance ✗. User taps any card to toggle. **NOT toggle switches** — use card boxes matching the direction/worktype button pattern.

Deal-breakers: 5 tappable card-style rows (full-width boxes with ✕ indicator when active, empty border when inactive). ALL pre-selected by default (unpaid work/spec work, no benefits offered, return-to-office mandate, requires relocation, travel required). "Return-to-office mandate" hidden if user selected On-site. User taps to deselect. The ✕ indicator (not ✓) reinforces that these are things being rejected/skipped. **NOT toggle switches** — use card boxes matching the quick check pattern.

**Step 3: Profile Generation & Resume Verification**

**Background Profile Generation**: Profile generation (PROFILE Claude call) is triggered immediately after resume parsing completes, rather than waiting for the user to reach Step 3. OnboardingPage orchestrates this: as soon as `triggerBackgroundParse()` succeeds, `triggerBackgroundProfileGen()` fires using whatever context and preferences exist at that moment (defaults for any fields the user hasn't filled yet). A phase ticker runs in OnboardingPage (4 phases x 5 seconds), and status is passed down to Step4Profile via `backgroundProfile`, `backgroundGenStatus`, and `backgroundGenPhase` props.

This means the profile may already be complete by the time the user finishes preferences and arrives at Step 3. The parallel architecture — resume parse, profile generation, and preference collection all overlapping — minimizes idle wait time.

- If profile generation completed during Step 2: the user sees their profile immediately on arrival
- If profile is still generating: the progress bar picks up from the current phase (no restart)
- If parse is still running: show "Waiting for resume parsing to complete..." with progress
- If parse failed: show retry option before profile can generate
- "Regenerate" button re-runs profile generation with the user's final (possibly updated) preferences

A confirmation screen presents personal info and education extracted from the resume — the user reviews and corrects parsed data here (name, email, phone, location, education entries). Required fields: name, email, phone, location. Optional: LinkedIn URL, portfolio URL (validated for HTTP/HTTPS). Education entries are editable inline (add/remove/modify). There is no standalone resume review step — verification happens alongside profile review.

**Late-arriving parse data**: Step4Profile has sync effects that update local `personal` and `education` state when props change (parse results arriving after component mount). This fixes a race condition where rushing through preferences could leave the confirm screen with empty personal fields — the data fills in as soon as parsing completes, even if the component is already mounted.

~~"Edit resume text" was removed in Run 007~~ — users can "Upload different" to start with a new file if needed. The confirm screen focuses on correcting parsed data, not re-editing raw text.

Claude AI (PROFILE prompt) analyzes the resume, context, and preferences to generate a structured professional profile: discipline, seniority level, hard skills list, differentiators, achievements, target direction, gaps to strengthen, avoid terms, and professional domains.

Profile fields stream in progressively as Claude generates them, reducing perceived wait time. A progress indicator shows 4 phases: reading resume, identifying strengths, mapping experience, building profile.

The user can edit discipline and seniority via dropdowns. Target direction is editable inline. A "Regenerate" button re-runs the full profile generation. "Looks good" confirms and advances.

**Post-Onboarding Celebration**

After completing all onboarding steps (target state: resume → preferences → profile → search → analysis; deep dive removed), a celebration screen plays before advancing to the dashboard. This marks the transition from setup to active job hunting. Deep-Dive Q&A is then available as an optional tier-jump activity from the dashboard. The auth gate appears at dashboard entry for unauthenticated users.

## 16. Decisions

1. **Resume re-upload**: "Upload different" button on confirm screen. ~~"Edit resume text" removed in Run 007~~ — re-editing raw text added complexity without proportional value. Users who need changes should upload a new file.
2. **Profile streaming**: Chunked JSON for MVP (parse partial responses client-side). Upgrade to SSE post-MVP.
3. **Demographics skip**: Soft nudge ("Completing demographics improves your results") but allow skip. Downstream features work without demographics but produce better results with them.
4. **Background parse architecture**: Resume parsing runs in background during preference collection. Profile generation also runs in background, triggered immediately after parse completes — it does not wait for the user to reach Step 3. This eliminates 10-30s idle wait and reduces abandonment. Trade-off: resume verification is deferred to the profile review screen (Step 3) rather than being a standalone step between upload and preferences. The background profile uses default preferences for any fields not yet filled; "Regenerate" re-runs with final values.

### GS-ONB-48: Server-Bound Session Nonce at Auth Wall

> As a System, I want the post-auth session identifier to be bound to a server-side record, so that an attacker cannot forge or replay a session nonce that only passed a format check (v2 behavior).

**Depends on:** none (spec-ahead; auth wall lives at dashboard entry, not inside onboarding)

**Entry state:** Transition — user completes onboarding and arrives at the auth wall at dashboard entry
**Verifiable by:** Complete onboarding → at auth wall, sign in → backend creates a server-side session record and issues a cookie whose nonce is recognized on subsequent calls; replay a nonce that matches the format but has no server-side record → 401 (backend PRD §8.3, decision 17).

**Acceptance Criteria:**

**Acceptance Criteria:**

- The session cookie issued at the auth wall uses the `__Host-` prefix where subdomain sharing is not required

# Onboarding Input Fields — Requirements

Working requirements doc for all user input fields across the onboarding screens.

**Global notes:**
- All screens auto-save to encrypted session on exit (survives refresh).
- Back/Next navigation available on all screens (except no Back on Resume).
- Screen names are internal — users see a continuous flow, not labeled steps.

**Control types:**
- **Cards** — for choices that deserve attention (full phrases, consequential). Slows the user down intentionally.
- **Pills** — for simple attributes/tags. Quick to scan, low friction, keeps the user moving.
- **Dropdown** — for long option lists that don't need to be visible all at once.

**Data contracts:** Each screen has a "Downstream data contracts" table showing which features consume its fields. See [DATA-CONTRACTS.md](../04-architecture/DATA-CONTRACTS.md) for the rules on wiring, ownership, and verification.

## Resume

> **Depends on:** None (entry point to the app)

### Control 1: File Upload

Drag-and-drop zone. Selecting or dropping a file begins processing immediately.

- **Label:** "Drag & drop your resume here"
- **Secondary label:** "or click to browse"
- **Accepted formats:** PDF, DOCX, TXT, MD
- **Max file size:** 10 MB
- **After selection:** Shows selected filename. Brief success animation (e.g., checkmark + "Got it") plays before advancing — minimum ~1s visible even if processing is instant, so the transition doesn't feel jarring.

### Control 2: Paste Text (alternate tab)

Textarea for pasting resume content directly.

- **Label:** Tab labeled "Paste Text"
- **Placeholder:** "Paste your resume text here..."
- **Min length:** 50 characters
- **Submit button:** "Parse resume →"

### Returning user (resumeRaw already exists)

- "Continue with existing resume" (primary action)
- "Upload different" (secondary — clears and shows upload mode)
- ~~"Edit resume text"~~ (REMOVED in Run 007 — users use "Upload different" instead)

### Trust & disclosure

- **Trust signals:** "Processed locally first", "Encrypted in your browser", "No data stored by AI"
- **Disclosure:** "Your resume will be sent to Claude for processing. Claude does not store your data."

### Hidden control: Cloudflare Turnstile (v3)

Invisible browser-verification widget. Not a visible form field — no label, no placeholder, no user interaction in the normal path.

- **Mount:** IntroScreen (first arrival per browser session)
- **Site key:** `NEXT_PUBLIC_CF_TURNSTILE_SITE_KEY` (public)
- **Mode:** Invisible (managed challenge). No checkbox or image grid shown.
- **Token destination:** `CF-Turnstile-Response` request header on the first anonymous `POST ${API_BASE_URL}/claude` call (PARSE) and on the first PROFILE call. Token is cached in component state for the session; never persisted to `SessionData` or localStorage.
- **Failure path:** If widget script fails to load or backend returns 403 on Siteverify, surface recoverable UI ("Please complete the challenge to continue" → "Try again"). Do not show raw 403.
- **Rate-limit path:** On 429 from Cloudflare per-ASN rate limit, surface "We're seeing heavy traffic from your network. Try again shortly." Do not auto-retry aggressively.

### Exit gate

Valid file uploaded or pasted text meets min length. After success animation completes, advances to Direction.

### City selector interaction

- **Data source:** Bundled US city list (~30k cities, all incorporated places). Format: "City, State" (e.g., "Austin, TX"). Zero API dependency.
- **Desktop:** Combobox — user types to filter, dropdown appears after 2+ characters. Arrow keys to navigate, Enter to select, Escape to dismiss. Max 8 results shown at a time.
- **Mobile:** Full-screen search sheet — slides up from bottom, full-width input pinned to top, scrollable results below. Results large enough for comfortable targets (min 44px height). X or swipe-down to dismiss.
- **Breakpoint:** Switch from inline dropdown to full-screen sheet at viewport width <= 640px.
- **Filtering:** Case-insensitive, matches anywhere in string (e.g., "aus" matches "Austin, TX" and "Thousand Oaks, CA"). Ranked by population so larger cities appear first.
- **Selection:** Single city. Selecting a result populates the input with the full "City, State" string and closes the dropdown/sheet.
- **No match:** Show "No cities found" message when filter returns 0 results.

If no match found, show a "Can't find your city? Enter it manually" link below the results area. Clicking it converts the input to a free-text field (placeholder: "City, State"). User types their city and proceeds.

## Profile Review

> **Depends on:** Resume parsing complete. Profile generation triggers in the background immediately after parse completes (does not wait for preferences). By the time the user finishes preferences and reaches this screen, the profile may already be ready. Personal/education fields sync reactively — if parse results arrive after this component mounts (user rushed through preferences), the fields populate automatically.

**Interaction philosophy:** AI fills, user curates. Subtractive where possible (chips), assisted where correcting (dropdowns), free text only where truly open-ended.

### Loading states

The screen has multiple states before the review is shown. Because profile generation runs in the background immediately after parse completes, the user may arrive at any point in this sequence:

- **Waiting** — resume parsing still in progress. Show spinner.
- **Parse error** — parsing failed. Offer retry.
- **Generating** — background profile generation in progress. Progress bar picks up from current phase (does not restart from phase 1). Step4Profile receives `backgroundGenPhase` prop.
- **Generation error** — generation failed. Offer regenerate.
- **Review** — background profile already complete. Show editable profile immediately (no generation delay).
