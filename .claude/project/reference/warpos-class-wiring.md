# WarpOS Class A/B/C wiring for updates and promotes

Phase 4K artifact (2026-04-30). Defines how the update / promote engines map their per-asset categories to the canonical Class A/B/C decision rule from `paths.decisionPolicy`.

## Source rule

`paths.decisionPolicy` (`.claude/agents/00-alex/.system/policy/decision-policy.md`) defines three classes:

- **Class A** — implementation, reversible. Decide directly, auto-apply.
- **Class B** — meaningful technical. Score against rubric, decide. Reviewer ack.
- **Class C** — strategic, irreversible, or business. Escalate with one recommendation, not a menu.

## /warp:update categories → class

The update engine (`scripts/warpos/update.js`) classifies every asset into one of 12 categories. Mapping:

| Category | Class | Rationale |
|---|---|---|
| `ADD_SAFE` | A | New file, no local conflict. Auto-apply. |
| `UPDATE_SAFE` | A | sha matches or local matches installed. Auto-apply. |
| `DELETE_SAFE` | A | Removed in target, local matches installed. Auto-delete. |
| `GENERATED_REBUILD` | A | File is regenerated by a deterministic script. Overwrite is safe. |
| `LOCAL_ONLY` | A | Local file outside framework. No-op. |
| `LOCAL_CUSTOMIZED` | A | mergeStrategy=keep_local. No-op. |
| `MERGE_SAFE` | B | mergeStrategy=three_way_markdown. Apply with reviewer (req-reviewer). |
| `RENAME_SAFE` | B | Source and target asset id matches by hash, dest path differs. Apply with reviewer. |
| `MIGRATION_REQUIRED` | B | Migration script attached. Apply with reviewer (output of plan() reviewed first). |
| `MERGE_CONFLICT` | C | Three-way merge fails — human resolution required. |
| `DELETE_CONFLICT` | C | Removed upstream but local diverged — human ack required before delete. |
| `RENAME_CONFLICT` | C | Same id but local diverged AND dest changed. Human resolution. |

Engine wiring lives in `update.js` `planClass()`.

## /warp:promote categories → class

The promote engine (`scripts/warpos/promote.js`) uses 9 categories. Mapping:

| Category | Class | Rationale |
|---|---|---|
| `FRAMEWORK_ADD` | A | New file in source, missing in target. Auto-add. |
| `FRAMEWORK_UPDATE` | A | Source differs from target on a tracked framework asset. Auto-update. |
| `FRAMEWORK_DELETE` | B | Target has it, source doesn't — needs human ack before deleting upstream (avoid accidental removal of WarpOS-only assets). |
| `GENERATED_IGNORE` | A | Skip — target regenerates. |
| `RUNTIME_IGNORE` | A | Skip — per-session state. |
| `PROJECT_IGNORE` | A | Skip — project-specific (e.g. jobzooka specs). |
| `MIGRATION_CANDIDATE` | B | Migration script — propagate if missing in target. Reviewer must confirm idempotence. |
| `TEMPLATE_REVIEW` | C | Per-project filled file — must be promoted as TEMPLATE only. Manual review. |
| `SECRET_BLOCK` | C | Looks like a secret — refuse. |

Engine wiring lives in `promote.js` `run()`'s classMap.

## Behavioral rules

1. **Class A is the only class that auto-applies.** No human in the loop, no reviewer dispatch.
2. **Class B requires reviewer dispatch.** The engine writes the plan but does not apply until the reviewer (typically `req-reviewer` for content drift, `compliance` for spec adherence) returns `pass` or `warn`.
3. **Class C halts.** The engine surfaces an `ESCALATE:` prefix message and refuses to apply. A separate explicit user action resolves each Class C item before the run can continue.
4. **Never silently downgrade.** A Class B item may NOT be auto-applied even if the engine "thinks it knows what to do." Bypass requires a flag (`--accept-class-b`) which is logged in events and surfaces in the run report.
5. **Class C bypass requires per-item ack.** No `--yes-to-all` for Class C. Each item must be acknowledged or dismissed individually with notes.

## Where the rule lives

- `scripts/warpos/update.js` `planClass()` — update mapping
- `scripts/warpos/promote.js` `run()` `classMap` — promote mapping
- `.claude/agents/00-alex/.system/policy/decision-policy.md` — source taxonomy
- This doc — the union; both engines must keep their internal maps consistent with what's documented here

When adding a new category to either engine, add a row to the appropriate table above first; the engines should match the doc, not the other way around.
